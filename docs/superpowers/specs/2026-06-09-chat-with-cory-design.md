# Design Spec: "Chat with Cory" — AI Career Assistant Widget

**Date:** 2026-06-09
**Status:** Approved design, pending implementation plan
**Repo:** `cory-carmen-portfolio` (private), branch `feature/ai-chat-widget`

---

## 1. Overview

An AI chat widget embedded directly into the live portfolio site
(`cory-carmen.vercel.app`). A visitor opens the widget, asks a question about
Cory's background, and receives a streaming, grounded answer with source
citations. The answer is generated by Claude using only Cory's own documents
(résumé, projects, bio), retrieved through a small Retrieval-Augmented
Generation (RAG) pipeline running in a Vercel serverless function.

The project is itself a portfolio piece: it demonstrates applied AI/RAG skills
on Cory's real website, integrated into his actual brand.

## 2. Goals & Success Criteria

**Goals**
- A working AI chat widget on the live portfolio that answers questions about Cory.
- Answers are grounded in Cory's real documents and cite their source.
- The feature is genuinely impressive to a general technical audience.
- The code reads well on GitHub and tells a clear story.

**Success criteria**
- Visitors can ask questions and get accurate, streaming, cited answers.
- The bot refuses to invent facts it cannot find in the source documents.
- A documented evaluation proves answer quality.
- The widget visually matches the existing site (Bootstrap + dark mode).
- Deploys with a single `git push` to the existing Vercel project.

## 3. Audience & Voice

- **Audience:** General technical reviewers (recruiters, hiring managers,
  fellow developers). Not aimed at one specific role — broadly impressive.
- **Voice:** Third person ("Cory has 3 years of experience..."). The bot is
  clearly an assistant answering *about* Cory, which reads as professional and
  recruiter-friendly.

## 4. Architecture

Everything lives in the existing portfolio repo. One repo, one Vercel deploy,
no new framework, no separate backend host, no CORS.

```text
cory-carmen-portfolio/
  index.html               # + chat widget container near end of <body>
  js/
    script.js              # existing site code (untouched)
    chat-widget.js         # NEW — vanilla JS chat UI (Bootstrap + dark-mode styled)
  api/
    chat.js                # NEW — Vercel serverless function (retrieval + Claude streaming)
  data/
    knowledge/             # NEW — source documents (markdown)
      resume.md
      projects.md
      bio.md
      links.md
    embeddings.json        # NEW — pre-computed embeddings (committed, generated by build script)
  scripts/
    build-embeddings.js    # NEW — one-time/local: turns data/knowledge/*.md into embeddings.json
  eval/
    questions.json         # NEW — tested question/expected-answer pairs
    run-eval.js            # NEW — scores bot answers, prints a report
  package.json             # NEW — dependencies for the function + scripts
  README-chatbot.md        # NEW — writeup of the feature for GitHub readers
  docs/superpowers/specs/  # this spec
```

**Why this shape:** the live site is plain HTML/CSS/JS + Bootstrap 5 (CDN) on
Vercel. Vercel serves serverless functions from `/api/*` for any static site,
so the backend drops in without changing the site's nature. The widget is
vanilla JS so it matches the existing codebase and needs no build step.

## 5. Components

Each component has one clear job and a simple interface.

- **`scripts/build-embeddings.js`** (build-time, runs locally)
  - Input: `data/knowledge/*.md`
  - Splits each document into chunks, calls the embedding API for each chunk,
    writes `data/embeddings.json` (array of `{ text, source, section, vector }`).
  - Run manually whenever the source documents change.

- **`data/embeddings.json`** (committed data)
  - The pre-computed vector index. Small (a few hundred KB at most).
  - Committed so the serverless function never needs to embed documents at runtime.

- **`api/chat.js`** (Vercel serverless function, Node)
  - Input: a user question (POST body).
  - Steps: embed the question → cosine-similarity search over `embeddings.json`
    → take top-K chunks → build a grounded prompt → stream Claude's answer back
    to the browser → include which sources were used (for citations).
  - Holds the API keys via Vercel environment variables (never sent to the browser).

- **`js/chat-widget.js`** (browser, vanilla JS)
  - Renders a floating chat button and panel, styled with Bootstrap classes and
    aware of the site's dark-mode state.
  - Shows suggested starter questions when empty.
  - Sends questions to `/api/chat`, renders the streaming answer, and displays
    citations beneath each answer.

- **`eval/run-eval.js`** (offline quality check)
  - Runs the question set in `eval/questions.json` through the retrieval +
    generation logic and reports how many answers are correct.

## 6. Data Flow

1. **Build (local, occasional):** `build-embeddings.js` reads
   `data/knowledge/*.md`, chunks and embeds them, writes `embeddings.json`.
2. **Ask (runtime):**
   - Browser widget sends the question to `/api/chat`.
   - The function embeds the question and finds the most relevant chunks in
     `embeddings.json`.
   - It builds a prompt containing those chunks + a system instruction (third
     person, grounded, no fabrication) and streams Claude's answer back.
   - The widget displays the streaming text and lists the cited sources.

## 7. Tech Stack

- **LLM:** Claude (Anthropic SDK). A fast, low-cost current model suits a chat
  widget — likely Claude Haiku 4.5 or Sonnet 4.6; the exact model is confirmed
  in the implementation plan. Third-person voice.
- **Embeddings:** OpenAI `text-embedding-3-small` (cheap, reliable, easy in Node).
- **Retrieval:** in-memory cosine similarity over `embeddings.json` — no database.
- **Frontend:** vanilla JS + Bootstrap 5 (already used by the site).
- **Hosting:** existing Vercel project; serverless function under `/api`.

## 8. Features

- **Streaming answers + citations** (core): answers stream token-by-token and
  show their source, e.g. "Source: Résumé → Experience".
- **Suggested starter questions:** clickable example prompts in the empty widget
  to solve the "blank box" problem.
- **Native branding:** the widget lives on Cory's real site, so name, headline,
  and links are already present; the widget matches the site's look.
- **Evaluation suite:** a tested Q&A set plus a script that scores answers,
  summarized in `README-chatbot.md`.
- **Grounded honesty:** when the answer is not in the source documents, the bot
  says so instead of inventing facts about Cory.
- **Cost & abuse protection:** capped response length, a simple per-session
  message limit in the widget, API keys server-side only, and a documented
  Anthropic spend cap.

## 9. Security & Cost

- **API keys** live only in Vercel environment variables, read by `api/chat.js`.
  They are never included in client code or committed to the repo.
- **`config.local.js`** (already gitignored) stays client-side and is unrelated
  to the LLM keys.
- **Cost:** embeddings are built once locally (pennies). Each chat is a fraction
  of a cent. Hosting is the Vercel free tier. A hard spend cap is set in the
  Anthropic console as a safety net.

## 10. Out of Scope (for now)

- Conversation memory / multi-turn follow-ups (each question answered fresh).
- A hosted vector database (not needed for small static data).
- User accounts or saving chat history.
- Analytics on what visitors ask.

These can be added later but are intentionally excluded to keep the first
version focused and shippable.

## 11. Risks & Mitigations

- **Prompt injection / abuse:** mitigated by a strict grounded system prompt,
  capped output, and per-session message limits. Worst case is a wasted few
  cents, bounded by the spend cap.
- **Hallucination about Cory:** mitigated by grounded-honesty instructions and
  the evaluation suite.
- **Touching the live site:** all work happens on `feature/ai-chat-widget`;
  `main` (and the live deploy) is untouched until Cory reviews and merges.

## 12. What Cory Provides (during implementation)

Templates will be supplied for each:
- Résumé text (`resume.md`)
- Project write-ups (`projects.md`)
- Short bio / about-me (`bio.md`)
- Links: GitHub, LinkedIn, résumé download (`links.md`)
